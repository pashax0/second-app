# Behavior

## Before changing anything

- Ask before making architectural decisions or introducing new patterns
- Prefer editing existing files over creating new ones
- When approach is unclear — propose first, wait for approval, then implement

## While implementing

- Do not add comments unless logic is non-obvious
- No unnecessary abstractions — solve the current problem only
- No dead code, no unused variables, no speculative additions
- Keep changes focused and minimal — don't "clean up while you're here"

## Communication

- Be concise — lead with the answer, not the reasoning
- If blocked or uncertain — ask, don't guess
- After each step: report what was done, then stop
