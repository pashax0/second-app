# План: Drops admin — расширение функционала

Статус: approved (in progress)

## Gap-анализ

Что админ **не может** сделать сегодня на `/drops`, `/drops/new`, `/drops/:id`:

- На детали дропа нельзя ничего изменить — это read-only страница ([DropDetail.tsx](../../apps/admin/src/pages/DropDetail.tsx)).
- При создании дропа нельзя задать дату публикации — она ставится автоматически на момент создания ([CreateDrop.tsx:80](../../apps/admin/src/pages/CreateDrop.tsx#L80)).
- При создании нельзя задать скидку на весь дроп, хотя бизнес такую механику предполагает ([business.md](../../.llm/context/business.md)).
- При создании нельзя задать индивидуальную цену товара в этом дропе.
- В списке выбора товаров нет поиска и фильтров — при росте фонда форма перестанет работать ([CreateDrop.tsx](../../apps/admin/src/pages/CreateDrop.tsx)).
- Превью миниатюр товаров в форме создания не показываются (выяснить причину при имплементации).
- После создания дропа нельзя добавить или убрать товар, изменить порядок, скорректировать цену.
- Активный дроп нельзя архивировать вручную, даже если все товары проданы или дроп нужно снять.
- Запланированный дроп нельзя удалить.
- На списке дропов нет поиска / фильтра по статусу, не видна дата планирования.
- При публикации нет превью «что произойдёт» (какой дроп архивируется, какой активируется).
- Все три дропа в seed отображаются как `(untitled)` — название не обязательно в БД, но обязательно в форме (несогласованность).

## Поведенческие решения

### 1. Редактирование дропа после активации
Активный дроп редактировать нельзя — что опубликовано, то опубликовано. Хочешь поправить — архивируй и пересобери. (Подтвердить или ослабить — открытый вопрос.)

### 2. Снятие товара из активного дропа
Возможно в любой момент. Снятый товар возвращается на склад и может быть включён в следующий дроп.

### 3. Скидка vs индивидуальная цена
В дропе одновременно могут существовать drop-wide скидка и индивидуальная цена для конкретного товара. Семантику приоритета (что показывается покупателю, если задано и то и другое) зафиксируем при имплементации соответствующего шага и продублируем в [product-lifecycle.md](../../.llm/context/product-lifecycle.md).

### 4. Удаление запланированного дропа
Удаление возможно **только** для дропа в статусе scheduled — он ещё никому не показан. Активные и архивные дропы — часть истории, не удаляются.

### 5. Название дропа
Название обязательно. Текущая несогласованность (БД допускает пустое, форма требует) — снимаем за счёт того, что пустое название перестанет быть валидным состоянием.

## План (что должен уметь пользователь после каждого шага)

**Milestone M1 — создание дропа**

| # | Что станет возможным | Acceptance |
|---|---|---|
| 1 | Найти товар в форме создания дропа поиском по названию / бренду / номеру и фильтрами по состоянию, размеру, признаку «возвращён со склада». | Ввести часть названия → список сужается. Поставить фильтр «Returned» → видно только возвращённые. |
| 2 | Увидеть превью фото товара в списке выбора (или явный плейсхолдер «нет фото»). | Открыть форму создания → у каждого товара миниатюра либо плашка «no photo». |
| 3 | Запланировать дроп на конкретную дату. | Выбрать дату завтра → сохранить → на детали дропа дата публикации = завтра. |
| 4 | Задать скидку на весь дроп (в процентах). | Создать дроп со скидкой 15% → на детали показано «−15%». |
| 5 | Задать индивидуальную цену конкретному товару внутри дропа. | Выбрать товар, ввести 1490 при базовой 1990 → на детали этого дропа цена 1490, у товара в фонде остаётся 1990. |
| 5a | Запретить повторное планирование одного товара в нескольких scheduled/active дропах. | Создать scheduled-дроп с товаром X → попытаться добавить X во второй scheduled-дроп → товар не виден в списке выбора; прямой INSERT в `drop_items` отвергается БД-триггером. |
| 5b | Планировать дроп на конкретный час, а не только день. | В форме создания указать «завтра 14:00» → на детали дропа scheduled_at показан с временем. |

**Milestone M2 — управление дропом после создания**

| # | Что станет возможным | Acceptance |
|---|---|---|
| 6 | Отредактировать название, описание, дату планирования и скидку запланированного дропа. | Открыть scheduled-дроп → нажать «Edit» → изменить поля → сохранить → значения видны после перезагрузки. |
| 7 | Добавить товар в запланированный дроп. | На детали scheduled-дропа выбрать товар из фонда → он появляется в списке позиций. |
| 8 | Убрать товар из запланированного или активного дропа. | На детали запланированного — товар пропадает. На детали активного — товар уходит обратно на склад и виден в списке `in_stock`. |
| 9 | Изменить индивидуальную цену товара внутри запланированного дропа. | На детали scheduled-дропа поменять цену строки → значение сохраняется. |
| 10 | Вручную архивировать активный дроп. | Нажать «Archive» на активном дропе → подтвердить → дроп переходит в archived, непроданные товары возвращаются на склад. |
| 11 | Изменить порядок товаров в запланированном дропе. | Поменять местами две позиции → после перезагрузки порядок сохранён. |

**Milestone M3 — список дропов и публикация**

| # | Что станет возможным | Acceptance |
|---|---|---|
| 12 | Найти дроп в списке поиском по названию и фильтром по статусу; видеть дату планирования. | Ввести запрос → список сужается. Выбрать фильтр «Scheduled» → видны только запланированные. В таблице есть колонка с датой планирования. |
| 13 | Удалить запланированный дроп. | На детали scheduled-дропа нажать «Delete» → подтвердить → дроп пропадает из списка. На активном/архивном кнопки нет. |

| 15 | Перестать видеть `(untitled)` в списке дропов. | Все существующие дропы имеют название; невозможно создать дроп без названия. |

**Milestone M4 — автопубликация (needs design)**

После каждого шага: stop, report, ждём подтверждения перед следующим. Шаги можно пропускать или откладывать по решению пользователя на месте — реализационный подход выбираем при старте каждого шага.

## Non-goals

- Push-нотификации при публикации дропа — отдельная задача (см. session.md → Open decisions).
- Массовое добавление товаров по фильтру — single-add справляется при текущих объёмах ([product-lifecycle.md](../../.llm/context/product-lifecycle.md) → «Что НЕ моделируем»).
- Журнал изменений (audit log) метаданных дропа.
- Сравнение / дашборд по дропам.
- Изменения в мобильном приложении — мобилка читает активный дроп и не знает о редактировании в админке.

## Риски

- **Скидка vs индивидуальная цена** — открытый вопрос приоритета (см. решение 3); закроем при имплементации M1.4–M1.5.
- **Редактирование активного дропа** — может появиться запрос «дай поправить опечатку»; зафиксировать решение при шаге 6.
- **Изменение порядка** — нужно убедиться, что после изменения позиций нет «дыр» и порядок воспроизводимый.
- **Согласованность названия** — снятие nullable-возможности коснётся существующих данных в seed; синхронизировать при шаге 15.

## Follow-up: доступ в админку

Всплыло при проверке шага 1: обычный покупатель (`test@test.test`, `is_admin=false`) спокойно открывает `/drops/new` в админке. Сабмит формы падает на RLS только потому, что Postgres не пускает не-админа прочитать свежесозданный `scheduled` дроп через `RETURNING` — но сама запись в `drops` проходит, write-полиси требуют только `auth.role()='authenticated'`. То есть и без формы любой залогиненный пользователь мобилки технически может писать в `drops` / `drop_items` / `product_images`.

Что сделать отдельной задачей (после M1):
- Закрыть вход в админку на клиенте — редирект на login, если `profiles.is_admin = false`.
- Привести write-полиси `drops`, `drop_items`, `product_images` к `is_admin()` — это уже паттерн всех lifecycle-RPC ([product-lifecycle.md](../../.llm/context/product-lifecycle.md)).
- Перепроверить остальные таблицы, у которых write — просто `auth.role()='authenticated'`.
