## Назначение

Одно рабочее место владельца магазина. Ежедневный цикл:
принять товар → сфоткать → завести в каталог → собрать дроп → опубликовать →
обработать заказы → повторить.

## Роли

Одна роль — **owner**. Multi-admin, RBAC, модерация — не в scope.
Админка не деплоится на хостинг: запускается локально с prod env.

## Функциональные области

### Товары (каталог)

- **Цель:** хранить фонд того, что может появиться на витрине.
- **Операции:** создать, редактировать (любого статуса), убрать с витрины (withdraw), вернуть на витрину (publish), списать (write-off), удалить (только in_stock без истории дропов; иначе soft delete).
- **Статусы и переходы** — [product-lifecycle.md](../../.llm/context/product-lifecycle.md). `products.status ∈ {in_stock, listed, sold, written_off}`. Изменения статуса — только через RPC.

### Дропы

- **Цель:** собрать текущую витрину из 15–20 товаров и опубликовать.
- **Операции:** собрать (status `scheduled`), опубликовать (`scheduled → active`), архивировать (`active → archived`), смотреть аналитику.
- **Границы:** одновременно `active` ровно один дроп. Публикация атомарна (RPC): архивировать старый + связанные products → `in_stock`; новый дроп → `active` + его products → `listed`.

### Заказы

- **Цель:** обработать COD-заказы от создания до доставки.
- **Операции:** список (new / confirmed / shipped / delivered / cancelled), детали, смена статуса, контакт покупателя.
- **Границы:** платёжной интеграции нет — только COD. CRM / трекинг-номера / SMS-оповещения — не в scope MVP.

### Резервации

- **Цель:** видеть активные корзины (что занято и когда освободится).
- **Операции:** список активных резерваций, принудительное истечение.

### Пользователи

- **Цель:** просмотр зарегистрированных, индивидуальный таймер корзины (доверенному — дольше, новому — меньше; правило из business.md).
- **Операции:** список, per-user override таймера.

### Настройки

- **Цель:** глобальный таймер корзины по умолчанию.

## Сценарии

**Ежедневный дроп:**
1. Принять новую партию (записать в `supply_lots` — для cost accounting)
2. Для каждой вещи: фото, замеры, condition, cost, list_price → создать товар в `in_stock`
3. Собрать дроп: добавить нужные `in_stock`-товары в `drop_items` дропа со `status='scheduled'`
4. Опубликовать (`activate_drop` RPC) — старый active дроп архивируется, его нераспроданные → `in_stock`; новый дроп → `active` + его товары → `listed`
5. В течение дня: обрабатывать заказы (confirm → ship); обрабатывать возвраты (`process_return` + инспекция); списывать повреждения (`write_off_product`)

**Ретроспектива прошлого дропа:**
- Открыть архивный дроп → продано, выручка, что «зависло» (вернулось в `in_stock` как Returned)

## Non-goals

- Multi-tenant / SaaS
- Платёжные интеграции (MVP — COD only)
- Публичный аналитический дашборд
- Push-кампании из UI (отдельная инфраструктура)
- Публичный хостинг админки

## Состояние реализации

| Область                                        | Статус     |
|------------------------------------------------|------------|
| Товары: list / create / edit / delete          | есть       |
| Товары: photo manager (reorder + remove + add) | есть       |
| Товары: withdraw / publish (mid-drop)          | есть (через RPC) |
| Товары: lifecycle (RPC + триггеры + view + supply_lots/returns/write_offs) | есть |
| Товары: composite-фильтры (Scheduled/Returned/In cart/Pending return) | есть       |
| Дропы: list / create / publish (через `activate_drop` RPC, атомарно) | есть |
| Дропы: edit / re-order items                   | нет        |
| Аналитика дропа (DropDetail)                   | есть       |
| Заказы                                         | нет        |
| Возвраты (returns + инспекция)                 | нет        |
| Резервации (отдельная страница)                | нет        |
| Пользователи                                   | нет        |
| Настройки (глобальный таймер)                  | нет        |

Привязка к коду — через `apps/admin/src/pages/*`. Здесь не дублируем имена файлов, чтобы roadmap не устаревал при переименованиях.

## Roadmap

В порядке приоритета (обсуждаемый):

1. **Orders page** — без этого магазин не операбелен за пределами Instagram-DM.
2. **Возвраты (returns) UI** — RPC и таблицы готовы (`process_return`, `complete_return_inspection`), нужна страница инспекции.
3. **Списания (write_offs) UI** — RPC `write_off_product` готова, нужна страница списаний и причин.
4. **Supply lots UI** — таблица есть, поле в ProductForm есть; нужна отдельная страница для создания партий.
5. **Settings + per-user timer** — реализация правила из business.md.
6. **Users page** — список и per-user overrides.
